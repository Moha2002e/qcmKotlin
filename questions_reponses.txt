Question 1 :
Concernant Intent, lesquelles sont vraies ?

A. A. Un Intent explicite cible une classe précise [CORRECT]
B. B. Un Intent implicite nécessite toujours un ComponentName
C. C. Les Intent implicites utilisent les IntentFilter [CORRECT]
D. D. Un Intent ne peut pas transporter d’objets complexes
E. E. Un Intent peut être utilisé pour démarrer un service [CORRECT]

Réponses correctes : A, C, E
Explication : Un Intent explicite cible bien une classe (A). Les Intent implicites reposent sur des IntentFilters pour trouver le composant adéquat (C). Depuis Android, il est courant d'utiliser des Intents pour démarrer des services (E), bien que startService soit moins utilisé au profit de JobScheduler/WorkManager, c'est techniquement vrai.

Question 2 :
Quelle(s) affirmation(s) sur PendingIntent sont correctes ?

A. A. Il permet à une application externe d’exécuter une action dans votre contexte [CORRECT]
B. B. Il est toujours lié à une Activity
C. C. Il est requis pour les notifications [CORRECT]
D. D. Il remplace un Intent
E. E. Il peut encapsuler un Broadcast [CORRECT]

Réponses correctes : A, C, E
Explication : Le PendingIntent est un jeton donné à une application étrangère (comme le NotificationManager ou l'AlarmManager) lui permettant d'exécuter une action avec les permissions de votre application (A). Il est indispensable pour rendre une notification cliquable (C). Il peut encapsuler un Intent de démarrage d'Activity, de Service ou d'émission de Broadcast (E).

Question 4 :
Concernant le cycle de vie d’une Activity

A. A. onCreate() est appelé une seule fois par instance [CORRECT]
B. B. onPause() garantit que l’activité est détruite
C. C. onStop() signifie que l’UI n’est plus visible [CORRECT]
D. D. onRestart() est appelé après onCreate()
E. E. onDestroy() peut ne jamais être appelé [CORRECT]

Réponses correctes : A, C, E
Explication : onCreate est le point d'entrée unique (A). onStop est appelé quand l'activité n'est plus visible par l'utilisateur (C). Le système peut tuer le processus sans appeler onDestroy en cas de besoin mémoire urgent, donc onDestroy n'est pas garanti (E).

Question 5 :
Laquelle (ou lesquelles) des affirmations suivantes est/sont correcte(s) concernant les constantes Intent.ACTION_* en Android ?

A. A. Elles représentent des actions standard reconnues par le système Android [CORRECT]
B. B. Elles sont principalement utilisées avec des Intents implicites [CORRECT]
C. C. Intent.ACTION_* permet de cibler directement une Activity précise
D. D. Elles peuvent être associées à des IntentFilter dans le manifeste [CORRECT]
E. E. Elles remplacent l’utilisation des ComponentName

Réponses correctes : A, B, D
Explication : Ce sont des chaînes standard (A) comme ACTION_VIEW ou ACTION_SEND, utilisées pour des Intents implicites (B) afin que n'importe quelle app compatible (déclarant ce filtre dans son manifeste, D) puisse y répondre.

Question 6 :
À propos des Fragment

A. A. Un fragment doit être attaché à une activité [CORRECT]
B. B. Un fragment possède son propre Context
C. C. Un fragment a un cycle de vie distinct [CORRECT]
D. D. Les fragments sont dépréciés
E. E. Un fragment peut survivre à une rotation [CORRECT]

Réponses correctes : A, C, E
Explication : Un Fragment vit nécessairement au sein d'une Activity (A) et possède un cycle de vie complexe lié mais distinct de celle-ci (C). Avec setRetainInstance(true) (bien que déprécié au profit des ViewModels), ou simplement via le mécanisme de restauration d'état, l'instance du fragment peut être conservée lors d'une changement de configuration (E).

Question 8 :
Laquelle (ou lesquelles) des affirmations suivantes est/sont correcte(s) concernant les expressions lambda en Kotlin ?

A. A. Une lambda est une fonction anonyme [CORRECT]
B. B. Une lambda peut être passée comme paramètre à une fonction [CORRECT]
C. C. Une lambda ne peut pas capturer de variables externes
D. D. Le type d’une lambda peut être inféré par le compilateur [CORRECT]
E. E. Une lambda ne peut pas retourner de valeur

Réponses correctes : A, B, D
Explication : Une lambda est bien une fonction sans nom (A), très souvent passée en paramètre (B). Kotlin infère son type (D). Contrairement à ce que dit C, elle peut capturer des variables (closure) et elle retourne la valeur de sa dernière expression (donc E est faux).

Question 9 :
Laquelle (ou lesquelles) des affirmations suivantes est/sont correcte(s) concernant les fonctions d’ordre supérieur en Kotlin ?

B. B. Une fonction d’ordre supérieur peut retourner une fonction [CORRECT]
C. C. Une fonction d’ordre supérieur doit obligatoirement être inline
D. D. Les fonctions d’ordre supérieur sont largement utilisées avec les lambdas [CORRECT]
E. E. Une fonction d’ordre supérieur ne peut pas être stockée dans une variable

Réponses correctes : A, B, D
Explication : C'est la définition même : prend des fonctions en paramètres ou en retourne (A, B). Elles sont conçues pour être utilisées avec des lambdas (D). Elles ne sont pas obligatoirement inline, bien que cela soit recommandé pour les performances.

Question 10 :
À propos de lateinit

A. A. Uniquement pour les types non-null [CORRECT]
B. B. Ne fonctionne pas avec les types primitifs [CORRECT]
C. C. Peut être utilisé avec val
D. D. Provoque une exception si non initialisé [CORRECT]
E. E. Compatible avec lazy

Réponses correctes : A, B, D
Explication : lateinit s'applique aux variables mutables (var) non-nulles (A). Elle ne supporte pas les types primitifs comme Int ou Boolean (B) car elle se base sur null en interne. Accéder à une propriété lateinit non initialisée lève une UninitializedPropertyAccessException (D).

Question 11 :
Différence entre lazy et lateinit

A. A. lazy est thread-safe par défaut [CORRECT]
B. B. lazy est uniquement pour val [CORRECT]
C. C. lateinit ne supporte pas les primitives [CORRECT]
D. D. lateinit est évalué à l’accès
E. E. lazy peut être réassigné

Réponses correctes : A, B, C
Explication : Lazy est une délégation pour les constantes (val) (B) qui initialise la valeur au premier accès de manière thread-safe (A). Lateinit est pour les variables (var) et ne marche pas, entre autres, avec les types primitifs (C).

Question 12 :
À propos des coroutines

A. A. Une coroutine n’est pas un thread [CORRECT]
B. B. Une coroutine peut être suspendue [CORRECT]
C. C. Une coroutine bloque toujours le thread
D. D. suspend n’indique pas l’asynchronisme [CORRECT]
E. E. Une coroutine survit à la mort du process

Réponses correctes : A, B, D
Explication : Une coroutine est un 'lightweight thread' géré par l'utilisateur, pas par l'OS (A). Elle peut être suspendue sans bloquer le thread sous-jacent (B). Le mot-clé suspend indique seulement qu'une fonction peut suspendre l'exécution, pas nécessairement qu'elle s'exécute sur un autre thread (D).

Question 13 :
Que fait exactement suspend ?

A. A. Rend la fonction asynchrone
B. B. Crée un thread
C. C. Permet la suspension sans bloquer [CORRECT]
D. D. Peut appeler d’autres fonctions suspendues [CORRECT]
E. E. Garantit un contexte IO

Réponses correctes : C, D
Explication : Le mot-clé suspend marque une fonction qui peut mettre en pause son exécution sans bloquer le thread courant (C). Seules les fonctions suspendues ou les coroutines peuvent appeler d'autres fonctions suspendues (D).

Question 14 :
Concernant runBlocking

A. A. Bloque le thread courant [CORRECT]
B. B. À utiliser en production Android
C. C. Utile pour les tests [CORRECT]
D. D. Remplace launch
E. E. Fonctionne uniquement sur Main thread

Réponses correctes : A, C
Explication : runBlocking démarre une coroutine et bloque le thread courant jusqu'à sa fin (A). C'est à éviter dans le code de production (pour ne pas figer l'UI), mais c'est très utile pour les tests unitaires ou les scripts (C).

Question 15 :
Quelle est la différence entre launch et async ?

A. A. launch retourne une valeur
B. B. async retourne un Deferred [CORRECT]
C. C. async nécessite await() [CORRECT]
D. D. launch bloque
E. E. async ne peut pas être annulé

Réponses correctes : B, C
Explication : Async est conçu pour calculer une valeur de manière asynchrone : il retourne un Deferred<T> (B) sur lequel on doit appeler await() pour récupérer le résultat (C). Launch retourne un Job et ne produit pas de résultat.

Question 16 :
Concernant Dispatcher.Main

A. A. Exécute sur le thread UI [CORRECT]
B. B. Peut être utilisé pour IO intensif
C. C. Requiert Dispatchers.Main.immediate pour éviter un post [CORRECT]
D. D. Est toujours disponible
E. E. Nécessite Android [CORRECT]

Réponses correctes : A, C, E
Explication : Dispatcher.Main utilise le thread principal (UI) d'Android (A). Il n'est disponible que sur les plateformes ayant une UI (Android, JavaFx...), donc E est vrai. Main.immediate est une optimisation pour exécuter la coroutine tout de suite si on est déjà sur le bon thread, évitant un dispatch inutile (C).

Question 17 :
À propos de l’annulation (cancel)

A. A. Elle est coopérative [CORRECT]
B. B. Elle stoppe immédiatement la coroutine
C. C. Les suspending functions la respectent [CORRECT]
D. D. Elle tue le thread
E. E. Elle propage aux enfants [CORRECT]

Réponses correctes : A, C, E
Explication : L'annulation en Kotlin est coopérative (A) : le code de la coroutine doit vérifier s'il est actif. Les fonctions de suspension standard vérifient l'annulation (C). L'annulation d'un parent entraîne celle de tous ses enfants (E).

Question 18 :

A. A. TimeoutException
B. B. IllegalStateException
C. C. CancellationException [CORRECT]
D. D. IOException
E. E. RuntimeException

Réponses correctes : C
Explication : L'annulation d'une coroutine se fait en jetant une CancellationException. C'est une exception spéciale qui est ignorée par les mécanismes de crash standard mais qui arrête la coroutine.
